# 文档提取工作流（非逆向方法）

本文档描述从源码提取文档的完整工作流。本项目仅使用静态代码阅读和工具辅助分析，不使用任何汇编逆向手段。

## 核心原则

- 源码是主要信息来源，Git 历史是设计意图的辅助来源
- 每个关键结论必须有可追溯的源码引用（文件路径 + 行号）
- 不确定的推断必须标注置信度（HIGH / MEDIUM / LOW）
- 先规划覆盖范围，再逐步提取，避免无序发散

## 阶段一：研究规划

在开始提取任何文档之前，先完成研究计划。

**步骤：**

1. 列出项目的主要系统和模块（参考目录结构、主要入口文件）
2. 在 `_meta/research-plan.md` 按系统划分研究域
3. 为每个研究域指定优先级（P0 核心 / P1 重要 / P2 补充）
4. 在 `_meta/documentation-gaps.md` 建立覆盖率追踪表

**产出：** 研究计划（含优先级和预计范围）、空白覆盖率表格。

## 阶段二：顶层结构扫描

快速建立对整个项目的宏观认知，不深入细节。

**方法：**

```bash
# 查看项目顶层目录结构
ls -la {SOURCE_ROOT}/

# 查找主要入口点
grep -r "Main\|Awake\|Start\|Initialize" {SOURCE_ROOT}/ \
  --include="*.cs" -l | head -20

# 统计各目录文件数量，了解代码分布
find {SOURCE_ROOT}/ -name "*.cs" | \
  awk -F/ '{print $NF}' | sort | uniq -c | sort -rn | head -30
```

**产出：** 顶层架构草图（Mermaid 图），填写到 `arch-01-overview.md`。

## 阶段三：静态代码阅读

针对每个研究域，系统地阅读源码并提取关键信息。

**调用链追踪方法：**

从入口方法开始，逐层向下追踪：

1. 找到系统的主要入口（`Initialize`、`Setup`、事件监听注册等）
2. 追踪每个公共方法的调用关系
3. 记录数据流向（哪些数据从哪里来，流向哪里）
4. 标注关键的条件分支和状态变化点

**信息记录格式：**

```
方法：ClassName.MethodName()
文件：{SOURCE_ROOT}/path/to/File.cs:156
作用：[一句话描述]
调用者：[调用此方法的上层方法]
被调用者：[此方法调用的下层方法]
关键逻辑：[核心逻辑的文字描述]
置信度：HIGH
```

**重点关注的信息类型：**

- 系统初始化顺序（依赖关系、启动顺序）
- 核心数据结构（主要类的字段和职责）
- 状态机（状态枚举、状态转换条件）
- 事件系统（事件发布者、订阅者、数据载荷）
- 外部依赖（网络、存储、第三方 SDK）

## 阶段四：Git 历史分析

利用 Git 历史挖掘设计意图和演化轨迹，补充静态代码阅读无法获取的上下文。

**常用命令：**

```bash
# 追踪单个文件的完整历史
git -C {SOURCE_ROOT} log --follow -p -- path/to/File.cs

# 查看某次提交的完整改动和提交信息
git -C {SOURCE_ROOT} show {commit-hash}

# 查找引入某个关键字的提交
git -C {SOURCE_ROOT} log --all -S "KeywordOrMethodName" --oneline

# 查看某段时间内的重大改动
git -C {SOURCE_ROOT} log --since="2024-01-01" --until="2024-06-01" \
  --oneline --stat | head -50

# 查找某个文件最近的修改者和原因
git -C {SOURCE_ROOT} blame path/to/File.cs
```

**Git 历史能回答的问题：**

- 「这个类为什么存在？」（查看创建时的提交信息）
- 「这个字段是什么时候加的，为什么？」（`git log -S`）
- 「这个系统经历了哪些重大重构？」（查看大型提交的 diff）
- 「有没有被删除又重新加回的逻辑？」（`git log --follow`）

## 阶段五：配置与数据文件解析

从非代码文件中提取结构化信息，通常比阅读代码更高效。

**目标文件类型：**

- 数据表文件（CSV、Excel、ScriptableObject）
- 配置文件（JSON、YAML、XML）
- 关卡/内容数据（二进制解析或文本格式）

**Python 脚本方法：**

在 `scripts/` 目录下编写提取脚本，输出结构化 JSON 或 Markdown 表格：

```python
# scripts/parse-config.py 示例结构
import json, csv

def extract_table(path):
    """从 CSV 数据表提取字段说明"""
    with open(path) as f:
        reader = csv.DictReader(f)
        return list(reader)

if __name__ == "__main__":
    data = extract_table("{SOURCE_ROOT}/Assets/Data/config.csv")
    # 输出为 Markdown 表格用于文档
```

## 阶段六：置信度评估与标注

对提取的每项关键信息进行置信度评估，防止将推断当作事实呈现。

**置信度等级：**

| 等级 | 含义 | 标注方式 |
|------|------|---------|
| HIGH | 直接从源码读取，无歧义 | `[置信度: HIGH]` |
| MEDIUM | 基于代码推断，有合理依据 | `[置信度: MEDIUM]` |
| LOW | 基于间接证据的假设 | `[置信度: LOW — 需验证]` |

**标注规则：**

- 所有标注 MEDIUM 或 LOW 的结论，必须说明推断依据
- LOW 级别的结论在文档中需要单独列出，便于后续验证
- 源码变更后，相关联的结论置信度需重新评估

## 阶段七：文档撰写

基于前几个阶段的笔记，撰写正式文档。

**撰写顺序：**

1. 先写架构概览（`arch-01-`），建立读者的整体认知框架
2. 再写核心机制（`core-`），深入关键流程
3. 然后写系统分析（`sys-`），覆盖各子系统
4. 最后补充参考资料（`ref-`）、操作指南（`how-`）等

**每份文档的必要元素：**

- 系统定位（一句话：这个系统做什么）
- 关键组件及其职责（表格或列表）
- 核心流程（Mermaid 序列图或流程图）
- 关键代码引用（文件路径 + 行号）
- 批判性分析（已知问题、技术债、改进方向）

## 阶段八：覆盖率更新

每完成一份文档，立即更新 `_meta/documentation-gaps.md` 中的覆盖率数据。

不要等到所有文档完成后再统一更新，保持覆盖率表格实时准确。
