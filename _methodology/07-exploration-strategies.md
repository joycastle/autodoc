# 探索策略分派

本文档定义 doc-research 在不同文档类型任务中的源码探索策略和 Explore agent prompt 模板。项目可根据技术栈特点调整探索重点和 prompt 模板。

## sys- / core- 任务（源码深度探索）

从「源码路径映射表」获取探索路径，启动 Explore subagent 并行阅读源码。

并行策略：
- 单路径任务：1 个 Explore agent
- 双路径任务：2 个并行 Explore agent
- 多路径任务：按路径数启动，上限 4 个

每个 Explore agent 的 prompt 模板：
「探索 {路径} 目录下的所有源码文件。重点记录：
 1. 核心数据结构和类型定义
 2. 主要函数/方法的调用链
 3. 状态机或状态转换逻辑
 4. 与外部系统的集成点（import/调用了哪些其他模块）
 5. 配置项和常量
 6. 异常处理和边界情况
 返回结构化的发现摘要。」

## tut- 任务（已有文档 + 源码实操路径探索）

教程不是从零探索源码，而是基于已有的 explanation 文档重新组织为学习路径。

探索分两步：
1. **读取派生源文档**：任务说明中标注了「派生自 XXX」，先读取对应的已完成文档
2. **验证实操路径**：用 Explore agent 验证教程中涉及的操作步骤在源码中确实可行
   - 如 tut-01 需验证：客户端项目能否按文档步骤搭建
   - 如 tut-02 需验证：架构导览中引用的入口文件是否存在

Explore agent prompt 模板：
「我正在写一篇教程，需要验证以下操作路径是否可行：
 {具体操作步骤列表}
 请检查相关文件是否存在，关键函数/配置是否如预期。
 返回每个步骤的验证结果和实际路径。」

## how- 任务（流程追踪探索）

操作指南关注「怎么做」，探索重点是操作流程而非系统架构。

探索策略：
1. **读取相关 sys-/ref- 文档**：获取系统背景知识
2. **追踪操作流程**：用 Explore agent 沿着「操作者视角」的调用链追踪
   - 从配置入口（配置文件、API 定义）开始
   - 追踪到业务逻辑（核心处理函数）
   - 到达生效点（服务端加载、客户端展示）

Explore agent prompt 模板：
「我正在写操作指南：{任务描述}。
 请从操作者视角追踪完整流程：
 1. 需要修改哪些配置文件/配置表？
 2. 修改后的生效路径是什么（热加载？重启？）
 3. 如何验证修改已生效？
 返回完整的操作流程链。」

## journey- 任务（端到端调用链追踪）

旅程文档需要从入口点开始，跨多个模块追踪完整的调用链路。

探索策略：
1. **读取沿途所有 sys- 文档**：获取各模块的内部知识
2. **链式追踪**：用多个 Explore agent 串联追踪调用链
   - Agent 1：追踪入口模块（前端触发 → API 请求）
   - Agent 2：追踪核心处理（后端 handler → 业务逻辑 → 跨模块调用）
   - Agent 3：追踪副作用（异步事件 → 订阅者响应 → 数据持久化）
   - Agent 4：追踪回路（后端推送 → 前端更新）
3. **故障分支**：在正常路径追踪完成后，识别关键故障分支点

Explore agent prompt 模板：
「我正在追踪端到端旅程：{旅程描述}。
 请从 {起点} 开始，追踪以下调用链：
 1. 当前模块收到请求/事件后，调用了哪些函数？
 2. 这些函数是否调用了其他业务模块的函数？（跨模块调用）
 3. 是否发布了异步事件？topic/channel 名称是什么？
 4. 是否写入了数据库？哪些表、什么数据？
 5. 是否向客户端发送了推送？推送内容是什么？
 6. 有哪些错误处理分支？（timeout、失败、重试）
 重点关注跨模块边界处的数据转换和错误传播。」

## theory- 任务（跨文档综合分析）

理论建构需要从多篇已有文档中提取共性模式，再回源码验证。

探索策略：
1. **读取所有相关文档**：任务说明中标注了涉及的文档数量和模块
2. **提取共性**：主线程分析多篇文档中的共同模式
3. **源码验证**：用 Explore agent 抽查 2-3 个代表性实例的源码实现
4. **演化追踪**（推荐）：调用 `/git-archaeology` 分析模式的历史演化
   - 该模式是一次性引入还是逐步演化？各模块的采用时间线如何？
   - 是否存在被替换的旧模式？重构里程碑在哪？
   - 将演化洞察纳入「决策指引」章节（为什么选了这个模式、何时该/不该用）

Explore agent prompt 模板：
「我正在分析跨模块的设计模式：{主题}。
 请深入阅读 {模块路径} 中与 {主题} 相关的实现。
 重点记录：
 1. 该模块如何实现这个模式（具体代码结构）
 2. 与其他模块的实现有何异同
 3. 该模式在此处的优势和局限
 返回详细的实现分析。」

Git 演化分析（theory- 推荐步骤）：
  调用 Skill("git-archaeology", args="code evolution for {相关模块路径列表}")
  从输出中提取：模式引入时间线、重构节点、废弃模式
  将「模式何时引入、为何演化」纳入文档的历史语境
