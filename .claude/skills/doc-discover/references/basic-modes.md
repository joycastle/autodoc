# 基础检测模式（模式 1-7）

本文档包含 doc-discover 基础检测模式的详细执行步骤。每个模式可独立运行，也可作为全量检测的一部分。

## 模式 1 — 模块扫描（modules）

检测源码中是否有新增模块未被文档覆盖。

```
扫描目标目录（从 $PROJECT_ROOT/_meta/source-path-mapping.md 读取，示例）：
  src/{backend-dir}/      → 每个子目录是一个业务模块
  src/{frontend-dir}/     → 每个子目录是一个客户端系统
  src/{api-dir}/          → 每个子目录是一个 API 服务
  src/{infra-dir}/        → 基础设施模块
  src/{tools-dir}/        → 工具模块

FOR 每个扫描到的模块:
  IF 该模块未出现在已覆盖清单且不在「已跳过模块」列表中:
    评估重要性（见 priority-type-tables.md「优先级判定」）
    确定文档类型（见 priority-type-tables.md「类型判定」）
    创建 create 任务
```

跳过阈值（不需要独立文档）：
- 文件数 < 3 的小目录（归入上级文档）
- 已被更高层文档充分覆盖的子模块
- 纯第三方库目录
- 测试目录

## 模式 2 — 漂移检测（drift）

检测已有文档对应的源码模块在文档写成后是否发生显著变更。优先使用 `/git-archaeology` 获取变更热点和演化洞察。

```
STEP A — 热点扫描（委托 git-archaeology）
  调用 Skill("git-archaeology", args="change hotspots in src/server/ src/client/ since {最旧文档日期}")
  获取按变更频率排序的模块列表
  与已有文档的源码路径映射交叉比对，识别高变更但已有文档的模块

STEP B — 逐文档漂移评分
  FOR 每篇已完成文档:
    获取文档对应的源码路径列表（从映射表）
    获取文档最后修改时间：git log -1 --format="%ai" -- {文档路径}
    （如果文档不在 git 中，使用文件系统 mtime）

    FOR 每个源码路径:
      统计文档之后的源码变更：
        git log --oneline --after="{文档日期}" -- {源码路径}
        计算 commit 数量和变更行数（git diff --stat {文档日期 commit}..HEAD -- {源码路径}）

    漂移评分：
      LOW（0-2 commits，< 50 行变更）→ 文档仍然新鲜
      MEDIUM（3-10 commits，50-200 行变更）→ 可能需要局部更新
      HIGH（> 10 commits 或 > 200 行变更）→ 需要审查

    IF 漂移评分 == HIGH:
      用 Explore agent 快速浏览变更（git diff --stat），判断变更性质：
      - 重构（重命名/移动但逻辑不变）→ 降为 MEDIUM，标记「路径可能需更新」
      - 新增功能 → 创建 improve 任务
      - Bug fix → 通常不需更新文档，跳过
      - 架构变更 → 创建 improve 任务，优先级提升一级
```

输出示例：
```
| 文档 | 源码路径 | 漂移 | commits | +/- 行 | 建议 |
|------|---------|------|---------|--------|------|
| sys-XX-{module-name} | {module-path}/ | HIGH | 15 | +320/-80 | improve: 新增功能 |
| sys-YY-{module-name} | {module-path}/ | LOW | 1 | +5/-3 | 无需更新 |
```

## 模式 3 — 类型平衡分析（balance）

检测 Diataxis 四象限的文档分布是否合理。

```
统计各类型文档数：
  tutorials（tut-）: 学习导向
  howto（how-）: 任务导向
  reference（ref-）: 信息导向
  explanation（arch-/sys-/core-/theory-）: 理解导向

健康基线（基于 Diataxis 框架推荐）：
  一个成熟项目应在四个象限都有产出
  最小健康阈值：每个象限至少有 2 篇实质性文档

识别缺口：
  IF 某象限文档数 < 2:
    标记为「类型缺口」
    基于已有的 explanation 文档，推荐可转化的文档主题：

    tutorials 缺口建议来源：
    - core-XX（{核心模块}） → tut-01: 新开发者环境搭建
    - arch-01（整体架构） → tut-02: 项目架构导览（新人 onboarding）
    - sys-XX（{核心系统}） → tut-03: {核心功能}开发教程

    howto 缺口建议来源：
    - sys-XX（{业务系统}） → how-XX: {常见操作}指南
    - ref-XX（API 文档） → how-XX: 新增 API 端点

    theory 缺口建议来源：
    - 跨 sys- 文档分析 → theory-XX: {跨模块共性模式}
    - 跨 sys- 文档分析 → theory-XX: {架构决策模式}

  为每个建议创建 create 任务（优先级 P2，类型为对应前缀）
```

## 模式 4 — 横切主题发现（themes）

识别跨多个模块反复出现、值得独立分析的设计模式或架构主题。

```
方法 A — 基于已有文档的文本分析：
  读取所有 $PROJECT_ROOT/docs/04_explanation/ 文档
  识别在 3+ 篇文档中重复出现的概念/模式：
    - 用 Grep 在文档中搜索高频技术术语
    - 关注反复出现的：设计模式、中间件、数据结构、通信协议

  常见横切主题候选（示例，按项目实际情况识别）：
    - 状态机模式（多个业务模块共用）
    - 数据访问模式（多个系统的数据层）
    - 消息路由（跨服务通信）
    - 配置热加载机制
    - 定时任务与生命周期管理
    - 安全与权限控制

方法 B — 基于源码的交叉引用：
  对源码路径映射表中的顶级模块，统计 import 图：
    grep -r "{模块名}" src/ 统计被引用次数
  高扇入模块（被 5+ 模块引用）如果尚未有独立 arch- 文档 → 候选

FOR 每个识别出的横切主题:
  检查是否已有文档覆盖（在 arch- 或 theory- 中）
  如果未覆盖且有足够素材（3+ 篇文档提及）:
    创建 theory- 或 arch- 类型的 create 任务
```

## 模式 5 — 深度审查（depth）

检测已有文档是否因源码模块增长而变得不够深入。

```
FOR 每篇已完成的 sys-/core- 文档:
  获取对应源码模块的当前规模：
    文件数：find {路径} -name "*.go" -o -name "*.cs" | wc -l
    总行数：wc -l（所有源文件）

  获取文档写成时的源码规模（如可从 research-log 中获取）
  或直接评估当前规模与文档深度的匹配度：

  规模-深度对照：
    小模块（< 10 文件，< 1000 行）→ 文档 1500-3000 字足够
    中模块（10-20 文件，1000-3000 行）→ 文档 3000-5000 字
    大模块（> 20 文件 或 > 3000 行）→ 文档 5000+ 字

  IF 当前文档字数明显低于建议范围（< 70%）:
    标记为「深度不足」
    创建 improve 任务，说明需要补充的方向
```

## 模式 6 — 角色视角分析（personas）

从不同项目角色的视角审视文档体系，发现「对某类用户有价值但当前缺失」的文档。

### 角色画像

```
FOR 每个角色画像:
  列出该角色的典型工作场景
  FOR 每个场景:
    检查现有文档是否能支撑该场景
    IF 不能 → 标记为缺口，确定文档类型和优先级
```

角色定义从项目的 `$PROJECT_ROOT/_methodology/` 目录加载（如 `$PROJECT_ROOT/_methodology/06-audience-roles.md`）。
IF 角色定义文件存在：读取并使用其中定义的角色列表和知识背景
IF 不存在：使用 $PROJECT_ROOT/_meta/task-queue.yaml（读写协议见 $PROJECT_ROOT/_meta/task-schema.md，写入后须重新生成 task-queue.md） 中的受益角色字段作为角色标签（无详细画像）

对于每个角色，按以下模板分析缺口：

```
#### {角色中文名}（{角色ID}）

日常场景：（从角色定义文件中获取「典型工作场景」）
- 场景 1 → 需要 {文档类型} 文档
- 场景 2 → 需要 {文档类型} 文档
- ...

缺口检测：
- 该角色最常用的文档类型是否有足够覆盖？
- 该角色的入门阶段是否有 onboarding 路径？
- 该角色的日常操作是否都有 step-by-step 指南？
```

对于带 `(onboarding)` 标签的角色，额外检测：
- 是否有面向该角色新入职的 tut- 或 how- 文档？
- 现有文档是否假设了该角色尚不具备的项目知识？

### 角色覆盖矩阵

```
构建「角色 x 场景」矩阵，标注每个场景的文档覆盖状态：

| 角色 | 场景 | 现有文档 | 覆盖度 | 缺口 |
|------|------|---------|--------|------|

覆盖度评级：
  FULL: 有专门文档直接覆盖该场景
  PARTIAL: 有相关文档但需要用户自行组合/推断
  NONE: 无任何文档覆盖

FOR 覆盖度 == NONE 的场景:
  创建对应的 create 任务
  优先级：该角色在团队中的占比越大 → 优先级越高

FOR 覆盖度 == PARTIAL 的场景:
  评估是否需要专门文档，还是在现有文档中补充即可
  IF 需要专门文档 → 创建 create 任务
  IF 补充即可 → 创建 improve 任务
```

## 模式 7 — 待验证点汇总（verify）

汇总 doc-research 在各文档中留下的待验证点，评估整体验证债务。

```
扫描所有已完成文档：
  用 Grep 搜索「置信度 MEDIUM」和「置信度 LOW」标注
  用 Grep 搜索 $PROJECT_ROOT/_meta/contracts/ 中的「待验证点」区

汇总统计：
  - 各文档的待验证点数量
  - 按类型分布（A 跨仓库 / B 运行时 / C 关联系统 / D 历史决策）
  - 已有 verify 任务覆盖的 vs 尚未覆盖的

FOR 尚未有 verify 任务的待验证点:
  IF 类型 A 或 C（可通过读取对端代码验证）:
    创建 verify 任务，优先级 P2
  IF 类型 B（需运行时验证）:
    记录但不创建任务（标注为「需人工验证」）
  IF 类型 D（需 git 考古）:
    创建 verify 任务，探索路径为 git log --follow
```

输出：
```
| 文档 | MEDIUM | LOW | 已有 verify 任务 | 新增 verify 任务 |
|------|--------|-----|-----------------|-----------------|
```
