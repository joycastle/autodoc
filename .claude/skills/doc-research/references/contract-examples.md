# Sprint Contract 参考

## Contract 要求

每条标准必须满足以下要求：
- 可验证：evaluator 能二元判定 pass/fail
- 聚焦内容覆盖和分析深度，而非格式（格式由通用检查表覆盖）
- 基于任务实际复杂度定制，不是泛泛而谈

Contract 写入 `$PROJECT_ROOT/_meta/contracts/T-{id}-contract.md`。

## 各类型 Contract 示例

### sys- 示例（sys-XX-{module-name}）

- [ ] 覆盖各子模块的核心数据模型和状态管理
- [ ] 分析关键数据的同步/一致性机制
- [ ] 说明与其他系统的集成点
- [ ] 识别数据访问模式和潜在性能风险

### tut- 示例（tut-01-dev-onboarding）

- [ ] 提供从零到可运行的完整步骤序列（前置条件 → 环境搭建 → 首次启动 → 验证）
- [ ] 每个步骤有明确的预期结果（学习者可自检是否成功）
- [ ] 不假设读者了解项目架构，仅引用必要的背景知识
- [ ] 包含至少 1 个「常见问题排查」段落

### theory- 示例（theory-01-state-machine-patterns）

- [ ] 从 3+ 个系统中提取状态机实例，对比异同
- [ ] 归纳出项目中状态机的共性设计模式
- [ ] 分析模式的优势、局限性和适用场景
- [ ] 为新系统采用该模式提供决策指引

### how- 示例（how-XX-{operation-name}）

- [ ] 步骤完整：从需求到上线的完整操作序列
- [ ] 每步引用具体文件路径和代码位置
- [ ] 包含配置示例（伪代码或精简的真实配置）
- [ ] 涵盖验证方法（如何确认操作成功）
