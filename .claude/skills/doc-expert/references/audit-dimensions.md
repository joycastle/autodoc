# 审计维度详述

## 维度 1 -- 角色阅读路径审计（paths）

模拟每种角色的典型阅读旅程，检查路径是否连贯。

```
IF $PROJECT_ROOT/_methodology/ 下存在角色定义文件：
  读取角色列表及其「典型工作场景」和「最常用文档类型」
ELSE:
  从 $PROJECT_ROOT/_meta/task-queue.yaml 的受益角色字段提取角色标签列表

FOR 每个角色:
  1. 确定该角色的入口文档（tut- 或概览型 arch-）
  2. 从入口文档的「相关文档」和「下一步」链接开始，递归追踪引用链
  3. 检查以下问题：

  问题 A -- 死胡同：
    文档没有「相关文档」或「下一步」章节，读者看完后不知道去哪
    -> 创建 improve 任务：补充相关文档引用

  问题 B -- 断裂链接：
    「相关文档」引用了不存在的文档
    -> 创建 create 或 improve 任务

  问题 C -- 深度不匹配：
    面向该角色的文档假设了超出其知识背景的前置知识
    例：面向 designer 的 how- 文档要求读者理解特定技术栈的底层细节
    -> 创建 improve 任务：降低技术深度或添加前置引用

  问题 D -- 覆盖盲区：
    该角色的某个典型工作场景没有任何文档支撑
    -> 创建 create 任务

  问题 E -- 入口缺失：
    该角色没有明确的入口文档（无 tut- 或入门级 how-）
    -> 创建 create 任务

  输出该角色的阅读路径图（文本形式）：
    入口 -> 第二层 -> 第三层 -> [断裂/死胡同/完整]
```

## 维度 2 -- 跨文档一致性检查（consistency）

检查不同文档对同一技术事实的描述是否一致。

```
抽样策略（避免全量比对的 O(n^2) 成本）：
  选择 3-5 个高频引用的技术概念（如核心通信协议、状态机流程、配置导出机制等）
  在所有文档中搜索对这些概念的描述

FOR 每个抽样概念:
  1. 收集所有文档中对该概念的描述段落
  2. 比较描述是否一致：
     - 术语是否统一（同一事物是否用了不同名称）
     - 数值是否一致（如超时时间、重试次数）
     - 流程是否一致（如步骤顺序、条件判断）
  3. 若发现矛盾：
     - 确定哪个文档的描述更接近源码事实（必要时用 Grep 验证）
     - 创建 improve 任务：修正不一致的文档

额外检查：
  - 术语表一致性：同一代码标识符在不同文档中的中文翻译是否统一
  - 源码引用一致性：同一文件在不同文档中引用的路径是否一致
```

## 维度 3 -- 叙事完整度评估（narrative）

评估文档库是否讲述了一个从架构到实现的完整故事。

```
检查三条主叙事线是否完整：

叙事线 A -- 架构全貌 -> 子系统深入：
  arch-01 (系统全景) -> 各 sys- (子系统分析) -> 各 core- (核心机制)
  检查：arch-01 中提到的每个子系统是否都有对应的 sys- 文档
  检查：sys- 文档是否引用回 arch- 文档作为上下文

叙事线 B -- 理解 -> 操作：
  sys- (理解系统如何工作) -> how- (如何操作该系统)
  检查：每个有操作需求的 sys- 文档是否有对应的 how-
  检查：how- 文档是否引用对应的 sys- 作为技术背景

叙事线 C -- 概念 -> 实践 -> 参考：
  theory- (设计模式) -> sys- (具体实现) -> ref- (查阅手册)
  检查：theory- 文档中引用的实例系统是否都有 sys- 文档
  检查：涉及配置的 sys-/how- 是否有对应 ref- 引用

对于断裂的叙事线：
  评估断裂的严重程度（影响多少角色、是否阻塞关键工作流）
  创建 create 或 improve 任务
```

## 维度 4 -- 文档索引和导航审计（index）

检查 docs/ 各子目录的 index.md 是否完整和可用。

```
FOR 每个 $PROJECT_ROOT/docs/ 子目录 (01_tutorials, 02_howto, 03_reference, 04_explanation):
  1. 读取 index.md
  2. 检查 index.md 是否列出了该目录下所有文档
  3. 检查是否有按角色或按主题的分类导航
  4. 检查新增文档是否已被添加到 index

IF index.md 缺失或严重过时：
  创建 improve 任务更新 index

检查跨目录导航：
  - 从 tutorials 到 howto 的过渡是否自然
  - 从 howto 到 reference 的查阅链接是否存在
  - 从 explanation 回到 tutorials/howto 的实践入口是否存在
```
