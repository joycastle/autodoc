# 文档质量二元评分表

达标阈值：**10/12（83%）**

每个检查项为 pass（1）或 fail（0），总分 = pass 数 / 12。

## 检查表定义

### 架构分析（arch-）

| # | 检查项 |
|---|--------|
| 1 | 系统定位段落存在（一段话说明系统在项目中的位置和职责） |
| 2 | Mermaid 架构图存在且节点数 ≤8 |
| 3 | 每个图表之后有中文说明段落 |
| 4 | 核心组件列表中每项均有源码引用（`src/...file:line`） |
| 5 | 核心流程有序列图或流程图（≤6 参与者 / ≤15 节点） |

## Contract 评估的类型适配

不同类型的 Contract 标准有不同的严格判定方式：

**sys-/core- Contract**：
- 「覆盖 X 机制」-> 必须有专门段落分析 X，不是一句话提及
- 「分析 X 风险」-> 必须有具体风险描述和源码证据
- 「说明 X 与 Y 的集成点」-> 必须有调用链或数据流描述

**tut- Contract**：
- 「提供 X 步骤」-> 步骤必须可直接执行，不是「参见文档」
- 「包含预期结果」-> 每步必须有可自检的预期输出
- 「不假设 X 知识」-> 相关概念必须内嵌解释或提供引用

**how- Contract**：
- 「步骤完整」-> 从头到尾无跳步，中间步骤不能省略
- 「包含配置示例」-> 示例必须与实际配置结构一致
- 「涵盖验证方法」-> 验证方法必须具体可执行

**journey- Contract**：
- 「追踪 X 到 Y 的完整链路」-> 序列图必须覆盖从 X 到 Y 的每一步，无跳步
- 「标注跨模块边界」-> 每个模块间调用必须标明协议和数据格式
- 「覆盖故障分支」-> 至少 2 个关键步骤的失败路径有具体分析

**theory- Contract**：
- 「从 N+ 系统提取」-> 必须有 N 个不同的具体实例分析
- 「归纳共性模式」-> 必须有明确的模式定义，不是罗列实例
- 「提供决策指引」-> 必须有可操作的判断框架，不是泛泛建议
